# Weather Dashboard - Приложение с использованием Kotlin Coroutines

## Описание проекта
Интерактивное приложение "Погодная панель" для Android, демонстрирующее работу с асинхронным программированием через Kotlin Coroutines и StateFlow. Приложение загружает и отображает данные о погоде (температуру, влажность, скорость ветра) с автоматическим обновлением каждые 10 секунд.

## Функциональность приложения
- 📊 Отображение текущей температуры с индикатором загрузки
- 💧 Показ влажности воздуха в процентах
- 🍃 Отображение скорости ветра в м/с
- 🔄 Ручное обновление данных по нажатию кнопки
- ⏱ Автоматическое обновление данных каждые 10 секунд
- ⚠️ Обработка ошибок и отображение сообщений об ошибках
- 🎨 Современный UI на Jetpack Compose с Material Design 3

## Технологии и библиотеки
- **Kotlin** - язык программирования
- **Jetpack Compose** - современный инструментарий для создания UI
- **Kotlin Coroutines** - асинхронное программирование
- **StateFlow & MutableStateFlow** - реактивные потоки данных
- **ViewModel** - компонент для хранения UI-данных
- **viewModelScope** - область видимости корутин
- **Dispatchers** - управление потоками (Main, Default, IO)
- **Material Design 3** - система дизайна

## Контрольные вопросы

### A) В чём разница между launch и async?

**launch**:
- Запускает корутину без возврата результата
- Возвращает `Job` для управления жизненным циклом
- Используется для "выстрелил и забыл" (fire-and-forget)

**async**:
- Запускает корутину с возвратом результата через `Deferred<T>`
- Позволяет получить результат через `await()`
- Используется для параллельных вычислений с получением результата

**Когда использовать:**
- `launch` - для обновления UI, сохранения данных, логирования
- `async` - для параллельной загрузки данных, вычислений

**Пример кода:**
```kotlin
// launch - без возврата значения
viewModelScope.launch {
    repository.saveData(data)
    updateUI()
}

// async - с возвратом значения
viewModelScope.launch {
    val deferredTemp = async { repository.fetchTemperature() }
    val deferredHumidity = async { repository.fetchHumidity() }
    
    val temperature = deferredTemp.await()
    val humidity = deferredHumidity.await()
    
    updateWeather(temperature, humidity)
}
```
### B) Что такое suspend функция?
**Suspend функция** - это функция, которую можно приостановить и возобновить позже. Она может вызываться только из другой suspend-функции или из корутины.
Может ли она вызываться из обычной функции?
- НЕТ! Suspend-функция может вызываться только:
- Из другой suspend-функции
- Из блока корутины (launch, async)
- Через runBlocking (в тестах)
Почему delay() не блокирует поток?
- delay() - это suspend-функция
- Она приостанавливает выполнение корутины, а не потока
- Поток освобождается для выполнения других задач
- Корутина возобновляется после истечения времени

```kotlin
suspend fun loadData() {
    delay(1000) // Не блокирует поток!
    // Код продолжит выполнение через 1 секунду
}
```
### C) Зачем нужны разные диспетчеры?
Диспетчеры определяют, в каком потоке будет выполняться корутина.
Таблица диспетчеров:
| Диспетчер | Для чего используется | Примеры задач |
|-----------|----------------------|---------------|
| **Dispatchers.Main** | UI операции, работа с Android UI | Обновление View, показ Toast, навигация |
| **Dispatchers.IO** | Операции ввода-вывода | Сетевые запросы, чтение/запись файлов, работа с БД |
| **Dispatchers.Default** | Тяжёлые вычисления | Сортировка, JSON парсинг, сложные расчёты |

Что будет, если выполнить тяжёлое вычисление на Dispatchers.Main?
-  UI зависнет и станет неотзывчивым
- Появится ANR (Application Not Responding)
- Пользователь не сможет взаимодействовать с приложением
- Система может завершить приложение

```kotlin
// ПЛОХО - блокирует UI
viewModelScope.launch {
    performHeavyCalculation() // UI зависнет!
}

// ХОРОШО - вычисления в фоновом потоке
viewModelScope.launch {
    val result = withContext(Dispatchers.Default) {
        performHeavyCalculation()
    }
    updateUI(result) // Возврат на Main поток
}
```

### D) Что произойдёт, если не обработать исключение в корутине?
Если не обработать исключение:
- Корутина завершится аварийно
- Исключение "всплывёт" и может завершить родительскую корутину
- В логах появится необработанное исключение
- Приложение может аварийно завершиться
Как корректно обрабатывать ошибки:
1. Использовать try-catch внутри корутины
2. Обновлять UI состоянием ошибки
3. Логировать ошибку для отладки
Зачем нужен try-catch внутри launch:

```kotlin
// Без обработки - опасно!
viewModelScope.launch {
    val data = repository.loadData() // Если упадёт - краш!
    updateUI(data)
}

// С обработкой - безопасно
viewModelScope.launch {
    try {
        val data = repository.loadData()
        updateUI(data)
    } catch (e: Exception) {
        showError(e.message) // Показываем ошибку пользователю
        _errorState.value = e.message
    }
}
```
### E) Как работает автоматическая отмена корутин?
Что такое viewModelScope?
- Это область видимости корутин, привязанная к ViewModel
- Автоматически управляет жизненным циклом корутин
- Все корутины, запущенные в viewModelScope, отменяются при уничтожении ViewModel
Когда корутины отменяются автоматически:
1. При вызове viewModelScope.cancel()
2. При уничтожении ViewModel (метод onCleared())
3. При отмене родительской Job
4. При возникновении необработанного исключения в родительской корутине

```kotlin
class WeatherViewModel : ViewModel() {
    private val _weatherState = MutableStateFlow(WeatherData())
    
    init {
        // Эта корутина автоматически отменится при уничтожении ViewModel
        viewModelScope.launch {
            while (true) {
                delay(10000)
                loadWeatherData()
            }
        }
    }
    
    override fun onCleared() {
        super.onCleared()
        // Все корутины в viewModelScope автоматически отменены
    }
}
```
#### Преимущества автоматической отмены:
✅ Предотвращает утечки памяти
✅ Останавливает фоновые задачи когда UI не виден
✅ Экономит ресурсы устройства
✅ Предотвращает обновление несуществующего UI
